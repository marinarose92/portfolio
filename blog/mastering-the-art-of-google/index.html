Mastering the Art of Google

If you’re teaching yourself programming, Google is going to become your best friend (and mentor).

I spend a lot of time solving problems at work, and one of the best ways to get a better handle on an IT problem is to Google it.

The first key to Googling effectively is persistence. When I’m trying to find an answer to a question I have, I could google several different queries and look at several different webpages - some of them I might come back to several times in an effort to process the information. It’s easy to get overwhelmed but you need to stick with it in the pursuit of knowledge.

Sometimes, people will be really surprised that I used Google to solve a particular problem. Knowing the right things to search and how to weed through all of the results is crucial to finding the information you need. With the right set of skills, Google can be the answer to pretty much anything. And the best part is, it’s instant! No endlessly refreshing, waiting for someone to reply on a forum. If you know what to find, you can probably find it.

Not knowing what to search for

Someone once said to me, “sometimes you don’t know what you don’t know”.

Especially in the early stages of learning to program, you may not be able to pinpoint exactly what the problem is. There are usually three ways to remedy this:

Try Google anyway

Chances are, someone may have asked as vague a question as the one you have now, so try just typing a question into Google’s search box the same way you might casually ask someone else:

Read everything.

If you’re not sure what to look for, chances are you don’t fully understand some basic concepts. Read through whatever tutorial you’re following or textbook you’re reading. Google “beginner’s guide to [your topic here]” or “all about [topic]”. Once you familiarize yourself with what you’re trying to tackle, you’ll probably have a bunch of new terms and concepts to search for.

Ask someone more experienced.

This is where chatrooms really excel, whether you’re using IRC, Slack or Discord. While it takes someone a while to parse through your code and comment on why a particular line might not be working, asking a more general question like, “hey, what is called when ….” is much more easily answered off the top of someone’s head. That way, when someone responds “it sounds like you want to use a callback function”, you’ll have some new terminology to Google.

Weeding the Results

Google returns a ridiculous number of results, scaling down the amount depending on how specific you are. To get better results, be specific.

I like to use a technique that I like to call “word vomit” where I just list all of the related things that may find me someone’s post who was in a similar situation to me.

One time, I wanted to include colored output to a command-line game I was working on in Repl.it. I wanted to optimize it for Repl.it because I knew I was going to be sharing a link with my friend, and I wanted them to be able to play my CLI game without having to download and run my script.

First, I tried: Ruby command line colors but I found that a lot of the code suggested by top rated posts weren’t working. Rather than try and troubleshoot the new code I found…

Take into account the big picture

I wasn’t just using colors in Ruby, I was using it with Ruby and Repl.it. Maybe some of the traditional way to get colors through a program worked with Ruby but not with Ruby on Repl.it. So I googled this phrase: replit ruby command line colors and immediately found something that helped me.

Just stack words on top of each other in your queries if you want to return websites that relate to your unique situation. When I was working on building this website, I prefixed a bunch of my search terms with: middleman ruby github pages … because I was using Middleman (a Ruby gem) and hosting it on Github pages. Sometimes picking something only related to Middleman wasn’t relevant to me, so including them both worked really well.
